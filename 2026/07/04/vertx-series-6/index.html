🌟 1. 从 Future 到 Actor：为什么还需要更高层的抽象？

上一篇我们掌握了 Future 编排——能把多个异步步骤串联或并行。但真实系统中，你常常面对这样的问题：

  • 一个”游戏房间”有自己的状态（玩家列表、当前回合、倒计时），多个请求并发操作它，怎么保证一致性？
  • 一个”订单”有生命周期（创建 → 支付 → 发货 → 完成），状态流转需要被保护。
  • 多个”设备连接”各自独立运行，互不干扰。

传统做法：加锁、用 ConcurrentHashMap、synchronized。

但在 Vert.x 的世界里，有更优雅的答案——Actor 模型。

而且你已经用过了：Verticle + Event Bus 天然就是 Actor。

🎯 2. Actor 模型简介

什么是 Actor？

Actor 模型是一种并发计算模型，核心思想极其简单：

每个 Actor 是一个独立的计算单元，拥有私有状态，只通过消息与外界交互。

三条铁律：

  1. 状态私有：Actor 的内部状态只有自己能访问
  2. 消息驱动：Actor 之间只通过发消息通信，不共享内存
  3. 顺序处理：每个 Actor 一次只处理一条消息

这意味着——无锁、无竞争、天然线程安全。

Verticle 就是 Actor

对比一下：

Actor 模型概念 Vert.x 对应
Actor Verticle
邮箱（Mailbox） Event Bus 地址上的消息队列
消息 Event Bus Message
私有状态 Verticle 的成员变量
顺序处理 Event Loop 单线程保证
创建子 Actor vertx.deployVerticle()

你不需要引入 Akka 或 Proto.Actor——Vert.x 本身就是一个轻量级 Actor 运行时。

🆚 7. 与 Akka Actor 的对比

特性 Vert.x Verticle Akka Actor
语言 Java / Kotlin / JS / … Scala / Java
消息投递 Event Bus（地址路由） ActorRef（直接引用）
监督策略 手动实现 内置完善
集群 Event Bus Cluster Akka Cluster
持久化 自行实现 Akka Persistence
状态恢复 自行实现 Event Sourcing 内置
学习成本 低 高
适合场景 IO 密集 + 中等并发实体 大规模分布式 + 复杂状态
依赖大小 ~7MB ~30MB+

选择建议：

  • 实体数在几千到几万级别，业务逻辑为主 → Vert.x Verticle 足够
  • 需要百万级 Actor、Event Sourcing、CQRS → 考虑 Akka
  • 已经在用 Vert.x 技术栈 → 不要为了 Actor 引入 Akka，用 Verticle 即可

🧠 8. Actor 模式的适用场景

适合用 Actor 的场景

场景 为什么适合
游戏房间 / 匹配大厅 每个房间独立状态，互不干扰
用户会话管理 每个用户一个 Actor，处理登录态
IoT 设备管理 每个设备一个 Actor，独立心跳
订单/工单状态机 状态流转需要保护，Actor 天然顺序
购物车 每个用户的购物车独立操作
聊天会话 每个对话一个 Actor，消息有序

不适合用 Actor 的场景

场景 为什么不适合
无状态 API 没有需要保护的状态，Actor 过重
批量数据处理 Actor 粒度太细，不如批量管道
简单 CRUD 数据库已保证一致性，没必要
全局统计/聚合 需要跨 Actor 汇总，违背隔离原则

🎯 10. 最佳实践

实践 说明
一个实体一个 Verticle 实例 保证状态隔离，避免并发冲突
消息定义清晰的 action 字段 类似 REST 的动词，易于理解
Actor 内部不要调用外部同步 API 阻塞 Event Loop = 整个 Actor 系统卡住
用 publish 广播事件，request 做命令 命令需要确认，事件只通知
控制 Actor 数量 用 LRU 池或超时销毁，防止内存泄漏
状态变更时发布事件 方便其他系统订阅（审计、统计、通知）

📌 11. 本章总结

本章你学到了：

  • Actor 模型的三条铁律：私有状态、消息驱动、顺序处理
  • Verticle = Actor：Event Loop 保证顺序性，Event Bus 提供消息通信
  • Verticle-per-Entity 模式：每个业务实体一个 Actor 实例
  • 状态机实现：利用 Actor 的顺序性保护状态转换
  • 监督策略：手动实现重启和错误处理
  • Actor 池：控制资源使用，LRU 淘汰冷 Actor
  • 适用场景判断：有状态 + 需要并发保护 = 用 Actor

现在你已经理解了 Vert.x 从 Event Bus 到 Future 到 Actor 的完整编程模型。但你是否好奇——底层到底是怎么运转的？为什么单线程的 Event Loop 能扛住高并发？ 下一篇，我们深入 Vert.x 的线程模型，从源码级别揭秘它的高性能秘密。
