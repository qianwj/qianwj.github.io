🎯 2. Future 与 Promise 的本质

Future：一个”未来的结果”

Future<T> 代表一个异步操作的最终结果。它要么成功（包含值 T），要么失败（包含异常）。

Promise：Future 的”写入端”

Promise<T> 是 Future 的生产者。你创建一个 Promise，把它的 Future 返回给调用方，然后在异步操作完成时调用 complete() 或 fail()。

一句话理解

Promise 是写入端，Future 是读取端。 就像一张支票：开票人（Promise）填金额，持票人（Future）等着兑现。

⚡ 4. CompositeFuture：并行执行多个异步操作

并非所有步骤都有依赖关系。比如下单前，”校验用户” 和 “检查库存” 可以并行执行。

all：全部成功才算成功

any：任一成功即算成功

join：等所有都完成（不管成败）

对比总结

方法 语义 失败条件
all 全部成功 任一失败即失败
any 任一成功 全部失败才失败
join 等全部完成 不会短路，全部执行完再回调

🆚 8. 对比其他异步方案

特性 Vert.x Future CompletableFuture Reactor Mono/Flux
线程模型 Event Loop 绑定 ForkJoinPool Scheduler 调度
操作符丰富度 够用（compose/map/recover） 较多 极其丰富
背压 无 无 有
学习曲线 低 中 高
与 Vert.x 集成 原生 需适配 需 vertx-rx 桥接
适合场景 IO 密集型编排 通用异步 流式数据处理

选择建议：

  • 纯 Vert.x 项目 → 用 Future，简单直接
  • 需要复杂流操作（窗口、合并、背压）→ 引入 vertx-rx（RxJava/Mutiny）
  • 与 Spring 混用 → CompletableFuture 做桥接

🎯 9. 最佳实践

实践 说明
方法签名返回 Future 让调用方自由组合，不要在方法内部 onSuccess
错误尽早失败 参数校验用 Future.failedFuture() 立即返回
recover 做降级，不吞异常 降级时 log 原始错误，方便排查
并行无依赖的操作 用 CompositeFuture.all，别串行等待
不要在 compose 里嵌套 compose 保持链式平铺，嵌套 = 回调地狱变种
避免在 Future 链中共享可变状态 用 map 传递数据，而不是外部变量
