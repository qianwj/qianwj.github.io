🌟 1. 为什么要自研 Gateway？

市面上有 Nginx、Kong、Spring Cloud Gateway、APISIX……为什么还要自己写？

场景 现有方案的痛点
定制路由逻辑 Nginx 用 Lua 写复杂逻辑很痛苦
与内部 Event Bus 打通 外部网关无法直接接入 Vert.x 集群
动态路由热更新 很多网关改配置需要 reload
协议转换 HTTP → Event Bus、gRPC → HTTP 等
极致性能要求 Spring Cloud Gateway 基于 WebFlux，有额外抽象层开销
学习目的 理解网关的本质，而不是只会配 YAML

本篇我们用 Vert.x 从零构建一个具备以下能力的 Gateway：

  • ✅ 反向代理（HTTP 转发）
  • ✅ 动态路由表（运行时热更新）
  • ✅ 限流（令牌桶）
  • ✅ 熔断（Circuit Breaker）
  • ✅ 重试策略
  • ✅ 请求/响应改写（Header 注入）
  • ✅ 链路追踪 ID 传递

测试环境

结果

网关 QPS 平均延迟 P99 延迟 内存占用
Nginx (proxy_pass) ~85,000 5.8ms 18ms ~15MB
Spring Cloud Gateway ~32,000 15ms 65ms ~300MB
我们的 Vert.x Gateway ~95,000 5.2ms 14ms ~80MB

分析：

  • 性能接近 Nginx（纯 C 实现），远超 Spring Cloud Gateway
  • 内存比 Spring 少 4 倍，比 Nginx 多（JVM 固有开销）
  • 优势在于：Java 生态 + 动态路由 + 与 Vert.x 服务天然整合

为什么能接近 Nginx？

  1. Netty 底层：和 Nginx 一样是 epoll/kqueue 驱动
  2. 零拷贝转发：Buffer 直接传递，不做多余序列化
  3. 连接复用：HttpClient 连接池，避免频繁建连
  4. 无锁路径：热路径上没有 synchronized（路由匹配是本地缓存读取）

🆚 11. 与主流网关对比

特性 Nginx Kong Spring Cloud GW Vert.x GW（本文）
语言 C Lua/Go Java Java
性能 ⭐⭐⭐ ⭐⭐ ⭐ ⭐⭐⭐
动态路由 reload 数据库驱动 配置刷新 Event Bus 实时推送
插件生态 丰富 非常丰富 较多 自行扩展
与 Java 服务集成 弱 中 强 极强（Event Bus）
运维复杂度 低 中（需 DB） 中 低
适合场景 通用反代/静态 API 管理平台 Spring 微服务 Vert.x 微服务 / 高性能网关

我们的 Gateway 适合：

  • 已经在用 Vert.x 技术栈的团队
  • 需要极致性能 + 灵活定制的场景
  • 需要与 Event Bus 集群打通的场景
  • 不想引入外部依赖（DB/Lua）的轻量化部署

🎯 12. 最佳实践

实践 说明
Gateway 多实例部署 充分利用 CPU 核心
路由匹配结果做本地缓存 避免热路径上走 Event Bus
连接池调大 HttpClient maxPoolSize 按后端服务数 × 预期并发设置
超时必须设置 连接超时 + 请求超时，避免慢后端拖死网关
熔断阈值按服务调整 核心服务放宽，非核心服务收紧
限流在网关层做粗粒度 精细限流交给各服务自己
链路追踪 ID 全链路传递 排查问题的生命线
响应体流式转发 不要在网关缓存完整响应体（大文件会 OOM）

📌 13. 本章总结

本章我们从零构建了一个生产级 API Gateway，它具备：

  • 反向代理：基于 HttpClient 的全异步转发
  • 动态路由：Event Bus 驱动，运行时热更新
  • 限流：令牌桶算法，按路由独立限流
  • 熔断：失败计数 + 半开探测，保护后端服务
  • 重试：指数退避，避免雪崩
  • Header 改写：链路追踪 ID 注入
  • 管理 API：RESTful 接口管理路由规则

这个项目综合运用了前面学的所有知识：Event Loop 保证高性能、Event Bus 实现组件通信、Future 编排异步逻辑、多实例部署充分利用 CPU。
