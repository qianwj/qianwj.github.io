🌟 1. 为什么需要集群？

到目前为止，我们所有代码都跑在单个 JVM 里。Event Bus 消息在进程内流转，Verticle 共享同一个 Vertx 实例。

但生产环境中，你会遇到这些问题：

问题 单机方案的局限
单机扛不住 CPU/内存/连接数到顶了
高可用 进程挂了，服务全停
地理分布 用户在不同地域，需要就近部署
服务拆分 不同团队负责不同模块，需要独立部署

Vert.x 的解决方案：集群模式——多个 Vert.x 实例组成集群，Event Bus 消息跨 JVM 透明传输。

关键特性：对应用代码完全透明。 你用 eventBus.send("address", msg) 发消息，不需要关心消费者在哪个节点——集群自动路由。

🎯 2. 集群模式的核心组件

Cluster Manager

Cluster Manager 是 Vert.x 集群的基础设施，负责：

  1. 节点发现：新节点上线/下线的检测
  2. Event Bus 路由：维护”哪个地址在哪个节点”的映射表
  3. 分布式数据：SharedData API 的跨节点实现
  4. 分布式锁：跨节点的互斥控制

可选的 Cluster Manager 实现

Cluster Manager 发现机制 适合场景 依赖
Hazelcast 组播/TCP 通用，功能最全 hazelcast
Infinispan JGroups Red Hat 生态，高性能 infinispan
Zookeeper ZK 注册 已有 ZK 集群的团队 curator
Ignite 组播/TCP 需要分布式计算 ignite

推荐： 大多数场景用 Hazelcast（默认、文档最多）。已有 ZK/Infinispan 基础设施的团队用对应方案。

🔄 4. 分布式 Event Bus 原理

消息路由过程

当你在 Node-1 执行 eventBus.send("order.create", data) 时：

关键步骤：

  1. 发送方查询 Cluster Manager 的路由表
  2. 找到目标节点后，通过 TCP 直连发送
  3. 消息使用 MessageCodec 序列化/反序列化
  4. 接收方在本地 Event Loop 触发 handler

send vs publish 在集群中的行为

模式 单机 集群
send 投递给一个本地 consumer 投递给集群中任意一个 consumer（轮询）
publish 投递给所有本地 consumer 投递给集群中所有节点的所有 consumer
request 等一个本地 consumer reply 等集群中任意一个 consumer reply

重要： send 在集群中依然是点对点（Round-Robin 选择一个目标），不会重复消费。

注册同步

当一个 Verticle 在 Node-2 上注册了 eventBus.consumer("order.create", ...)：

  1. Node-2 的 Cluster Manager 将 ("order.create", Node-2) 写入分布式注册表
  2. 所有节点收到通知，更新本地路由缓存
  3. 后续任何节点 send 到 "order.create" 都知道去找 Node-2

节点下线时，其注册的地址会被自动清理。

🛡 5. 分布式 SharedData

集群模式下，Vert.x 提供跨节点共享数据的 API。

分布式 Map（AsyncMap）

分布式锁（Lock）

分布式计数器（Counter）

使用建议

数据类型 适合 不适合
AsyncMap 会话存储、配置共享、缓存 高频读写（性能不如 Redis）
Lock 分布式任务调度、防重复处理 长时间持有（会阻塞其他节点）
Counter 全局计数、ID 生成 精确计数（网络分区时可能不一致）

⚠️ SharedData 适合轻量级场景。高频读写或需要持久化的场景，建议用 Redis。

🔍 8. 节点故障与恢复

故障检测

Cluster Manager 通过心跳检测节点存活：

Cluster Manager 默认超时 检测方式
Hazelcast 5s TCP 心跳
Infinispan 30s JGroups FD
Zookeeper session timeout Ephemeral Node

故障时的行为

当 Node-2 宕机：

  1. Cluster Manager 检测到 Node-2 心跳丢失（~5s）
  2. 路由表清理：所有 Node-2 注册的 Event Bus 地址被移除
  3. 后续消息路由：send("order.process") 只会发往 Node-1 和 Node-3
  4. SharedData 清理：Node-2 持有的锁自动释放

HA（High Availability）部署

Vert.x 支持 Verticle 级别的 HA——当某节点宕机时，其 Verticle 自动迁移到其他节点：

当节点挂掉时：

  1. 集群检测到节点离开
  2. 该节点上标记为 HA 的 Verticle 会在其他存活节点上重新部署
  3. 服务自动恢复

🐛 9. 常见问题与排错

① 节点无法发现彼此

症状： 启动后日志只显示 1 个成员。

排查：

常见原因：

  • 组播被防火墙/Docker 网络阻断 → 改用 TCP 发现
  • cluster-name 不一致 → 确保所有节点配置相同
  • 端口冲突 → Hazelcast 默认 5701，检查是否被占用

② Event Bus 消息发出去但对方收不到

排查步骤：

  1. 确认两个节点在同一个集群（日志中应显示 member joined）
  2. 确认 consumer 注册在 send 之前完成
  3. 检查 MessageCodec——自定义对象是否注册了 Codec
  4. 检查防火墙——Event Bus 集群通信用的端口（默认随机）是否放开

📊 10. 集群性能考量

Event Bus 跨节点 vs 本地

操作 本地 Event Bus 跨节点 Event Bus
延迟 <0.1ms 0.5-2ms（取决于网络）
吞吐 ~500,000 msg/s ~50,000 msg/s（单对节点）
序列化 无（对象引用传递） 有（需要 Codec）

优化原则：

  • 高频通信的 Verticle 尽量部署在同一节点
  • 跨节点传输的消息保持小体积
  • 大对象先存 Redis/S3，Event Bus 只传引用（ID/URL）

集群规模建议

节点数 注意事项
2-5 Hazelcast 默认配置即可
5-20 关闭组播，用 TCP 发现；监控 Hazelcast 内存
20-50 考虑分组（不同服务用不同集群）
50+ 不建议单集群这么大，拆成多个小集群 + gRPC 互联

🎯 12. 最佳实践

实践 说明
生产环境禁用组播 用 TCP 或 Kubernetes 发现
固定 Event Bus 端口 方便防火墙和监控
明确绑定 IP 多网卡环境避免绑错接口
奇数个节点 防止脑裂时无法选举
监控集群健康 定期检查成员数、Event Bus 延迟
小消息原则 跨节点消息控制在几 KB 以内
HA 只用于有状态 Verticle 无状态服务靠多实例即可
优雅停机 先从 LB 摘除，再关闭节点
集群 ≠ 数据库 SharedData 做缓存，不做持久存储
