🔍 3. 瓶颈定位：火焰图 + JFR

async-profiler 火焰图

怎么看火焰图：

  • 横轴 = 该方法占总采样的比例（越宽越耗资源）
  • 纵轴 = 调用栈深度
  • 找最宽的”平顶”——那就是瓶颈

JFR 持续采集

JFR 中重点关注的事件：

事件 用途
jdk.ExecutionSample CPU 采样，等同火焰图
jdk.ThreadPark Event Loop 阻塞定位
jdk.JavaMonitorEnter 锁竞争
jdk.ObjectAllocationSample 内存分配热点
jdk.GCPhasePause GC 停顿时长和频率
jdk.SocketRead / jdk.SocketWrite 网络 IO 耗时

用 JMC（JDK Mission Control）打开 .jfr 文件分析。

参数 作用

📊 9. 优化全过程回顾

轮次 优化内容 QPS 提升 累计提升
基线 未优化 51,263 - -
第一轮 消除 Event Loop 阻塞 78,432 +53% +53%
第二轮 减少序列化开销 112,890 +44% +120%
第三轮 GC 调优（G1→ZGC） 128,560 +14% +151%
第四轮 Vert.x 参数调优 152,340 +18% +197%
第五轮 系统级调优 198,760 +30% +288%

从 5.1 万到 19.9 万，提升近 4 倍。

🧪 10. 持续性能监控

优化不是一次性的，需要持续监控防止性能退化。

Vert.x Micrometer Metrics

关键监控指标

指标 含义 告警阈值
vertx_http_server_requests_seconds 请求延迟分布 P99 > 100ms
vertx_http_server_active_requests 当前活跃请求数 > 线程数 × 100
vertx_eventbus_pending Event Bus 待处理消息 持续增长
vertx_pool_queue_size Worker Pool 队列长度 > 0 持续
vertx_pool_usage_seconds 线程池使用时间 接近 100%
jvm_gc_pause_seconds GC 停顿 P99 > 50ms
jvm_memory_used_bytes 堆内存使用 > 85%

📌 13. 本章总结

本章你掌握了：

  • 基准测试方法论：wrk 全链路压测 + JMH 微基准
  • 瓶颈定位三件套：async-profiler 火焰图 + JFR 事件分析 + Metrics 监控
  • 五轮递进优化：
    1. 消除阻塞（+53%）
    2. 减少序列化（+44%）
    3. GC 调优 G1→ZGC（+14%）
    4. Vert.x 参数调优（+18%）
    5. 系统级调优（+30%）
  • 性能反模式：日志风暴、资源泄漏、大 JSON、CPU 密集阻塞
  • 持续监控：Micrometer + Prometheus + Grafana

核心心法： 不猜测，先测量；先消灭最大的瓶颈，再优化细节。
