🌟 1. 为什么在 Vert.x 里用 gRPC？

上一篇我们构建了一个 HTTP Gateway。但在内部微服务通信中，HTTP/JSON 有明显的短板：

痛点 说明
序列化开销大 JSON 文本解析比二进制慢 5-10 倍
没有强类型契约 接口变更容易出错，靠文档约束
不支持流式通信 HTTP/1.1 天生请求-响应，双向流很难
代码生成弱 OpenAPI 生成的代码质量参差不齐

gRPC 正好解决了这些问题：

  • Protobuf 序列化：体积小、速度快
  • 强类型 IDL：.proto 文件即接口契约，编译期校验
  • HTTP/2 传输：多路复用、双向流
  • 原生代码生成：一份 proto 生成多语言客户端/服务端

而 Vert.x 5.0 提供了自己的 gRPC 实现（vertx-grpc-server / vertx-grpc-client），不依赖 grpc-java 的 Netty 传输层，直接复用 Vert.x 的 HTTP/2 服务器。同时内置了 gRPC Transcoding 和 gRPC-Web 支持，无需中间代理。

🎯 2. Vert.x gRPC vs grpc-java：为什么要”重新造轮子”？

特性 grpc-java (grpc-netty) Vert.x gRPC
传输层 自带 Netty（与 Vert.x 的 Netty 冲突） 复用 Vert.x HTTP/2 Server
线程模型 自己的线程池，与 Event Loop 割裂 原生 Event Loop 集成
编程风格 StreamObserver 回调 Future + ReadStream/WriteStream
依赖大小 ~20MB（grpc-netty-shaded） ~2MB（仅 vertx-grpc）
Transcoding 需要 Envoy 或 grpc-gateway 内置支持，零代理
与 Event Bus 集成 手动桥接 天然互通
GraalVM Native 困难（大量反射） 友好

核心优势： Vert.x gRPC 不是 grpc-java 的包装，而是一个独立实现——直接在 Vert.x HTTP/2 上实现 gRPC 协议，保证与 Event Loop 的完美契合。

📊 11. 性能对比

测试场景

结果

方案 QPS P50 延迟 P99 延迟 内存
grpc-java (Netty) ~85,000 2.1ms 8ms ~250MB
Vert.x gRPC ~92,000 1.9ms 7ms ~120MB
Vert.x Transcoding (HTTP/JSON) ~65,000 2.8ms 12ms ~120MB
纯 HTTP/JSON (Vert.x Web) ~95,000 1.8ms 6ms ~80MB

分析：

  • Vert.x gRPC 性能略优于 grpc-java，因为少了一层线程切换
  • Transcoding 有 ~30% 的开销（JSON 序列化/反序列化），但换来了兼容性
  • 纯 HTTP/JSON 最快（没有 Protobuf 编解码），但没有类型安全

🎯 12. 最佳实践

实践 说明
Proto 文件单独仓库管理 多团队共享契约，版本控制
所有 RPC 加 HttpRule 即使暂时不用 Transcoding，也方便后续开放
错误码用 gRPC Status NOT_FOUND / INVALID_ARGUMENT 等标准码
大数据用流式 列表查询用 server streaming，别一次返回 10 万条
截止时间（Deadline）必须设 防止慢调用堆积，客户端设 5s 超时
Protobuf 向后兼容 只新增字段，不删除/修改已有字段编号
Transcoding 仅对外暴露 内部服务间走纯 gRPC（性能更好）

🆚 13. 微服务通信选型：HTTP vs gRPC vs Service Proxy

Vert.x 生态提供了三种服务间通信方式，各有适用场景。

Service Proxy 是什么？

Service Proxy 是 Vert.x 提供的Event Bus RPC 代码生成方案。你定义一个 Java 接口，Vert.x codegen 自动生成：

  • Event Bus 消息发送/接收的样板代码
  • 客户端代理类（像调本地方法一样调远程服务）

生成后的使用方式：

本质上是对 Event Bus request/reply 模式的语法糖。

三者全方位对比

维度 HTTP (Vert.x Web) gRPC Service Proxy
传输层 HTTP/1.1 或 HTTP/2 HTTP/2 Event Bus（本地/集群）
序列化 JSON（文本） Protobuf（二进制） JSON over Event Bus
IDL/契约 OpenAPI（可选） .proto 文件（强制） Java 接口 + 注解
代码生成 可选 必须 必须
跨语言 ✅ 任何语言 ✅ 多语言 ❌ 仅 JVM
流式通信 ❌（需 WebSocket） ✅ 四种流模式 ❌ 仅请求-响应
性能（同 JVM） ⭐⭐ 有网络开销 ⭐⭐ 有网络开销 ⭐⭐⭐ 无网络，纯内存
性能（跨节点） ⭐⭐ ⭐⭐⭐ ⭐⭐ 依赖集群管理器
服务发现 需要注册中心 需要注册中心 Event Bus 集群自动路由
浏览器兼容 ✅ 原生 ⚠️ 需 Transcoding ❌
调试便利 ⭐⭐⭐ curl 直接调 ⭐⭐ 需 grpcurl ⭐ 需要 Vert.x 环境
运维复杂度 低 中（需管理 proto） 低

分层原则：

通信层 方案 说明
外部 → Gateway HTTP / gRPC + Transcoding 浏览器和第三方兼容
服务 ↔ 服务 gRPC 跨节点、强类型、高性能
服务内部模块 Service Proxy / Event Bus 同 JVM，零开销，最简单

什么时候不用 Service Proxy？

场景 原因 替代方案
需要跨语言 Service Proxy 只支持 JVM gRPC
需要流式传输 Service Proxy 仅支持请求-响应 gRPC Stream
高频跨节点调用 Event Bus 集群性能不如直连 gRPC gRPC
需要精细的超时/重试/熔断 Service Proxy 不提供这些 gRPC + 拦截器

什么时候优先用 Service Proxy？

场景 原因
同 JVM 内多 Verticle 协作 零序列化，极快
小团队 / 纯 Vert.x 项目 不需要维护 proto 文件，Java 接口即契约
快速原型开发 定义接口就能用，无需学 Protobuf
同集群少量节点通信 Event Bus Cluster 够用，不想引入额外协议

📌 14. 本章总结

本章你学到了：

  • Vert.x gRPC 是独立实现：复用 HTTP/2 Server，不依赖 grpc-java
  • Protoc 插件：生成 Vert.x 风格的 Future/Stream 代码
  • 四种通信模式：一元、服务端流、客户端流、双向流
  • Transcoding：同一端口同时服务 gRPC 和 HTTP/JSON 客户端
  • 拦截器：认证、日志、监控的切面
  • 与 Event Bus 结合：gRPC 做外部接入，Event Bus 做内部路由
  • Service Proxy vs gRPC vs HTTP：分层选型，各取所长
  • 选型原则：对外用 HTTP/gRPC+Transcoding，服务间用 gRPC，服务内部用 Service Proxy
