🌟 1. 从请求-响应到实时推送

前几篇我们处理的都是”客户端发请求、服务端回响应”的模式——无论是 HTTP 还是 gRPC 一元调用。但有些场景天然不适合这个模型：

  • 股票行情：服务端要主动推送价格变化
  • 聊天消息：消息到达时立即送达，不能靠客户端轮询
  • 协同编辑：多个用户的操作要实时同步给所有人
  • 游戏状态：帧同步数据需要低延迟双向通信
  • IoT 设备：设备持续上报数据，服务端随时下发指令

这些场景的共同特点：连接长期保持，双向实时通信。

WebSocket 就是为此而生的协议。而 Vert.x 基于 Netty 的实现，能轻松处理 10 万+并发连接。

📊 6. 10 万连接：性能优化

内存估算

单个 WebSocket 连接占用：

  • TCP Socket Buffer：~8KB（读+写）
  • Netty Channel 对象：~2KB
  • 业务层状态（userId, subscriptions）：~1KB
  • 合计：~11KB / 连接

10 万连接 ≈ 1.1GB。一台 4GB 堆内存的 JVM 可以轻松承载。

🔄 8. 多节点推送：跨实例消息路由

单机 10 万连接已经很强了。但当连接数超过单机容量，或者需要高可用时，就需要多节点部署。

问题来了：用户 A 连接在节点 1，推送请求到了节点 2，怎么办？

方案一：Event Bus Cluster

业务系统发送 push.user 消息到 Event Bus，集群模式下会自动路由到正确的节点。

优点： 代码不用改，Event Bus 自动处理。
缺点： 集群管理器（Hazelcast）有一定开销，节点数多时效率下降。

方案二：Redis Pub/Sub 做跨节点广播

优点： 简单可靠，Redis 吞吐高。
缺点： 所有节点都收到所有消息（浪费带宽），需要自行过滤。

方案三：一致性哈希路由

优点： 精准路由，无浪费。
缺点： 节点变化时需要迁移连接。

推荐选择

集群规模 推荐方案
2-5 节点 Event Bus Cluster（最简单）
5-20 节点 Redis Pub/Sub（稳定可靠）
20+ 节点 一致性哈希 + 直连（性能最优）

🎯 10. 最佳实践

实践 说明
连接时立即认证 不要接受匿名连接，防止资源耗尽攻击
心跳双向确认 服务端和客户端都要检测对方存活
消息格式统一 约定 {type, data, timestamp} 结构
背压处理 writeQueueFull 时丢弃或断开慢消费者
离线消息存储 用户断线期间的重要消息存 Redis/DB，重连后补推
批量广播分批次 避免单次循环阻塞 Event Loop
压缩大消息 启用 Per-Message Compression
监控连接数 实时监控，设置容量告警
优雅断开 发 Close Frame + 原因码，便于客户端区分
