在前面两章中，我们搭建了开发环境并运行了第一个 Vert.x 项目。

从本章开始，我们将进入 Vert.x 最核心的概念之一：

Verticle —— Vert.x 中部署、隔离、调度的最小执行单元。

你可以把 Verticle 理解成：

  • 类似 Node.js 中的“模块”
  • 类似微服务内部的“服务单元”
  • 类似 Java 应用中的一个“独立组件”

Verticle 是构建 Vert.x 应用的基础模块，本章将带你系统掌握它的原理、生命周期、部署方式、线程模型等关键知识。

🌟 1. 什么是 Verticle？

Verticle（组件/单元）是 Vert.x 程序的可部署单元，每个 Verticle：

  • 有独立的生命周期（start/stop）
  • 跑在 Event Loop 或 Worker 线程中
  • 可以独立部署、独立扩容
  • 通过 EventBus 与其他 Verticle 通信

🌱 2. Verticle 的生命周期

Verticle 有 2 个核心生命周期方法：

✔ 1）start()

Verticle 部署时执行。

通常在这里：

  • 创建 HTTP 服务器
  • 注册事件总线消费者
  • 初始化数据库连接
  • 启动定时任务

✔ 2）stop()

Verticle 被卸载时执行。

通常在这里：

  • 关闭资源（DB、Redis）
  • 取消定时器
  • 清理业务逻辑

🧱 3. Verticle 的分类

Vert.x 提供两大类 Verticle：

🔹 1）标准 Verticle（Event Loop Verticle）

默认类型。

特点：

  • 运行在 Event Loop 线程
  • 不允许长时间阻塞
  • 响应速度极快

适合：

  • HTTP 请求处理
  • WebClient 调用
  • EventBus 事件处理
  • 轻量业务逻辑

不要做：

  • 阻塞 IO
  • 大量 CPU 密集型操作
  • 睡眠、等待锁、循环卡死

因为 Event Loop 线程一旦阻塞，整个系统吞吐立即下降。

🔹 2）Worker Verticle（阻塞 Verticle）

用于执行阻塞任务。

部署方式：

特点：

  • 运行在线程池 Worker Pool
  • 可以执行阻塞操作
  • 不影响 Event Loop

适合：

  • JDBC 阻塞查询（非 Reactive 查询）
  • 文件 IO、大量 CPU 计算
  • 需要睡眠、线程等待的业务逻辑

🧩 5. 多实例 Verticle：并发能力的核心

默认情况下，一个 Verticle → 一个 Event Loop 线程。

但如果你要提升处理能力，可以创建多个实例：

这意味着：

  • 同一个 Verticle 会跑在 4 个 Event Loop 线程上
  • 同时处理更多并发请求
  • 更适合 CPU 较多的服务器

Vert.x 默认 Event Loop 线程数 = CPU 核数 × 2

因此在多核机器上多实例非常有用。

🧠 6. Verticle 的线程模型

Vert.x 的线程模型是异步高性能的基础。

🔸 Event Loop Thread（事件循环线程）

  • 用于处理所有非阻塞业务逻辑
  • 每个 Event Loop 线程必须“永不阻塞”
  • 如果阻塞会导致整个线程 stall（卡死）

这是 Vert.x 性能强大的核心原因。

🔸 Worker Thread（工作线程）

  • 来自 Worker Pool（默认 20）
  • 允许阻塞逻辑
  • 不会影响 Event Loop

🔸 Context：Vert.x 的“线程上下文”

每个 Verticle 都运行在一个“Context”中。

重要特性：

  • 同一个 Verticle 实例永远运行在同一个 Event Loop
  • 避免了传统多线程的同步问题（不需要 synchronized/lock）
  • 构建 Actor-like 架构非常容易

🎯 10. 实战建议（最佳实践）

✔ 保持 Event Loop 上的代码“轻量”

规则：

Event Loop 上的代码永不上锁、不睡眠、不做耗时操作。

✔ 阻塞逻辑必须放到 Worker Verticle 或 executeBlocking

✔ 每种类型的业务写成独立 Verticle

比如：

  • API Verticle（HTTP 接口）
  • Service Verticle（业务）
  • DB Verticle（数据访问）
  • Timer Verticle（定时任务）

这样可扩展性、可维护性非常强。

✔ 合理设置实例数量

  • API Verticle：多实例
  • DB Verticle：按连接池大小决定
  • Worker：根据 Worker Pool 数量
